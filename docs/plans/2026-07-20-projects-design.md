# Projects Design

**Date:** 2026-07-20

**Status:** Approved

## Summary

Projects is a new QLayer Quick Tool that groups one user-selected local folder with related Codex chats, preferred development ports, detected localhost servers, and two predefined development actions. The feature extends the existing Chat Shortcuts, Voice Flow, and Localhost Manager capabilities without rebuilding or weakening them.

QLayer remains a local-first coordination layer. It does not launch, stop, restart, suspend, or otherwise manage development processes directly. Start and Stop Development are fixed requests delivered to a user-selected Codex chat. Localhost Manager remains the source of truth for whether the expected ports are actually active.

## Goals

- Let the user create, edit, inspect, and delete Projects.
- Associate one local folder with each Project.
- Link relevant Codex chats to a Project.
- Suggest recent chats when their Codex working directory matches the Project.
- Configure labeled preferred ports with explicit roles and strictness.
- Relate detected localhost servers to a Project only when the folder identity is reliable.
- Send fixed Start Development and Stop Development requests to a chat selected for each action.
- Fall back to copying the fixed message and opening the selected chat when Codex App Server cannot deliver it.
- Add a `Chats | Projects` mode switch to the existing Voice Flow destination selector.

## Non-goals

- Activity or history feeds.
- Usage Insights.
- Saved Prompts.
- Add-ons, plugins, or a marketplace.
- Direct process launch, termination, suspension, or restart.
- Arbitrary command or shell execution.
- Editable development-action prompts.
- A special, authoritative, or exclusive Control Chat.
- Network traffic inspection, interception, or proxying.
- Reading credentials, tokens, Codex authentication files, browser cookies, or chat contents.

## Project model

The persisted model is strongly typed and versioned. Chat links retain enough local display metadata to remain understandable when a thread is no longer in the recent-chat response.

```ts
export type ProjectPortRole = "frontend" | "backend" | "fullstack" | "database" | "other";

export type ProjectPort = {
  id: string;
  label: string;
  role: ProjectPortRole;
  port: number;
  strict: boolean;
};

export type ProjectChatLink = {
  threadId: string;
  displayName: string;
  linkedAt: string;
};

export type Project = {
  id: string;
  name: string;
  rootPath: string;
  rootIdentity: string;
  linkedChats: ProjectChatLink[];
  preferredPorts: ProjectPort[];
  lastSelectedChatId?: string;
  createdAt: string;
  updatedAt: string;
};
```

`rootIdentity` is an opaque, case-insensitive fingerprint generated natively from the user-selected absolute folder path. It uses the same identity algorithm as Localhost Manager. It is not a credential and is not sent over the network.

## Validation

- Project name: trimmed, required, maximum 80 characters.
- Root path: selected with the native Windows folder dialog, must be an existing absolute directory when created or changed.
- Project identity: required and generated by native code from the selected folder.
- Chat thread IDs: canonical Codex UUID form, unique inside the Project.
- Chat display names: trimmed, required, maximum 80 characters.
- Port labels: trimmed, required, maximum 40 characters.
- Ports: integers from 1 through 65535 and unique inside the Project.
- Port role and strictness: validated against explicit enums and booleans.
- Malformed stored entries are dropped safely while valid siblings remain available.

## Persistence

Projects are stored locally under a versioned key such as `qolayer.projects.v0`. Storage parsing is defensive and returns an empty collection plus a recoverable warning when the envelope is malformed. No cloud synchronization or telemetry is added.

Deleting a Project removes only its QLayer configuration. It never removes the folder, Codex chats, Localhost Manager data, or running processes.

## Project creation and editing

`New Project` opens a dedicated editor view rather than compressing fields into the list. The editor includes:

1. Project name.
2. A native `Choose folder` action.
3. Linked-chat selection from saved and recent Codex chats.
4. Preferred-port rows with label, role, numeric port, and strictness.
5. `Cancel` and `Save Project` actions.

Editing uses the same dedicated view. Changing the folder recalculates the opaque root identity and refreshes chat/server suggestions.

## Project list

The Projects navigation item uses a Tabler-style folder icon in the existing sidebar. The list view follows the approved compact mockup:

- Header: `Projects`, Project count, and a small add button.
- Rows use dividers rather than independent cards.
- Each row shows Project name, shortened folder location, linked-chat count, and expected-port summary.
- Reliable running ports use restrained success indicators; missing ports use muted indicators.
- Empty state: `No projects yet.` with one concise `New Project` action.

## Project detail

The detail view contains:

- Back navigation.
- Project name and shortened root path.
- Current expected-port summary.
- A three-dot menu containing `Edit Project` and `Delete Project`.
- `Start Development` and `Stop Development` actions.
- A Localhost section with expected ports and verified states.
- A Chats section with linked chats and a compact `Link` action.

The three-dot menu replaces the standalone pencil icon shown in the initial mockup.

## Project deletion

Choosing `Delete Project` opens a confirmation dialog that names the Project and states that only its QLayer configuration will be removed. The dialog provides `Cancel` and a destructive `Delete Project` action. Destructive styling is restrained but unambiguous. Keyboard focus remains inside the dialog until it closes.

## Chat association

Chats can be linked from existing saved destinations or from recent Codex chats. Recent-chat discovery is extended to return an opaque root identity derived from the thread working directory; it does not expose the raw directory to the renderer through this discovery path.

When a recent chat root identity matches the Project root identity, it is shown first as a suggestion. Suggestions are never linked automatically. A linked chat can be opened using the existing canonical `codex://threads/<id>` deep link.

There is no Control Chat. Start and Stop Development always require a chat selection. `lastSelectedChatId` only highlights a convenient default and has no authority over later actions.

## Localhost association and verification

Localhost Manager already returns an opaque `projectId` for reliably resolved development servers. A server is verified as related when its `projectId` equals the Project `rootIdentity`.

Preferred-port states are:

- `Running`: expected port is active and its root identity matches.
- `Missing`: no active listener was detected on the expected port.
- `Conflict`: the expected port is active but reliably belongs to another Project.
- `Unverified`: the expected port is active but its Project identity is unavailable.
- `Checking`: a restrained refresh is in progress after a development action.

A port match alone never establishes ownership. Database and other non-openable listeners can still be shown as active, but QLayer preserves Localhost Manager's rule that only classified development-server URLs can be opened.

For a strict port, an alternate port remains a mismatch. For a non-strict port, a reliably matched server on a different port can be shown as an alternate without being reported as the configured port.

## Development actions

`Start Development` and `Stop Development` are fixed, structured actions. The user cannot provide a command or edit the message template.

Each action follows this sequence:

1. The user selects one linked chat.
2. QLayer opens or focuses that exact Codex/ChatGPT chat.
3. Native code builds the fixed message from the validated Project name, folder, action enum, and port structures.
4. QLayer uses the verified Codex/ChatGPT application runtime and the documented Codex App Server stdio protocol.
5. It initializes the session, resumes the selected thread, and starts a turn with the fixed message and Project working directory.
6. Any approvals remain visible and controlled inside Codex/ChatGPT. QLayer never grants them automatically.
7. Localhost Manager performs restrained verification of the configured ports.

Start Development asks Codex to inspect the repository, start the appropriate development environment, prefer or enforce the structured ports, and avoid changing unrelated processes.

Stop Development asks Codex to stop only development servers reliably associated with the selected Project and configured ports, and to request confirmation when ownership is ambiguous.

## App Server fallback

App Server is the primary documented integration. If the action cannot be delivered before the turn is accepted:

1. Native code returns the already-built fixed message as a fallback payload.
2. QoLayer writes that message to the clipboard using write-only clipboard permission.
3. QoLayer opens the selected Codex chat.
4. QoLayer displays `Message copied. Paste it in Codex to continue.`

QoLayer never reports that development started or stopped merely because a message was sent or copied. Actual state comes from Localhost Manager.

## Voice Flow integration

The existing Voice Flow selector remains the entry point and keeps its current push-to-talk behavior. A segmented `Chats | Projects` switch is added at the top.

### Chats mode

- Preserves `Current chat` and the existing individually saved destinations.
- Preserves numeric selection behavior.

### Projects mode

- First shows Projects.
- Choosing a Project shows its linked chats.
- A back action returns to the Project list.
- Selecting a chat starts the existing targeted Voice Flow.
- Empty state: `No linked chats`.
- Numeric shortcuts apply to the currently visible list.

QoLayer remembers only the last selector mode as a convenience. It does not persist an authoritative destination. Releasing the push-to-talk shortcut while selecting continues to cancel the flow and restore audio safely.

## Error handling

- Missing/deleted folder: show `Project folder unavailable` and allow editing.
- Deleted/unavailable chat: keep the Project usable, mark the chat unavailable, and allow unlinking.
- App Server unavailable before delivery: invoke the approved clipboard fallback.
- App Server failure after the turn is accepted: show an action error without duplicating the request.
- Clipboard failure: keep the fixed message visible in a selectable read-only field and open the chat.
- Localhost inspection failure: preserve the last snapshot and show a non-destructive warning.
- Process disappearance during refresh: recalculate state from the newest snapshot.
- Duplicate action request: reject it while the first Project action is still active.

## Security and privacy

- The native action command accepts a typed action enum and structured Project data, never arbitrary prompt or command text.
- Only the verified Codex or ChatGPT desktop application runtime can be used.
- No authentication files, tokens, browser cookies, or chat contents are read.
- Recent-chat parsing continues to read metadata only.
- No network listener, proxy, telemetry, or cloud sync is introduced.
- Folder access is user-initiated through the native dialog.
- Tauri permissions are limited to folder selection and clipboard writing; clipboard reading is not granted.
- QLayer does not answer Codex approval requests.

## Accessibility and motion

- User-facing text remains at least 14 px.
- All icon-only controls have accessible labels and tooltips.
- Menus and dialogs are keyboard reachable.
- Status is never conveyed by color alone.
- View and selector transitions use the existing short easing and respect `prefers-reduced-motion`.

## Testing

- Type validation and storage migration/recovery tests.
- Project-to-localhost status derivation tests.
- Recent-chat root-identity parsing tests in TypeScript and Rust.
- Fixed action-message snapshot tests in Rust.
- App Server framing, handshake, delivery, timeout, and fallback tests using fake stdio data.
- Project list/detail/editor/delete component tests.
- Voice selector parsing, two-level navigation, keyboard selection, cancellation, and existing Chats-mode regression tests.
- Existing Voice Flow, Localhost Manager, shortcut, audio, focus, tray, and settings suites must remain green.

## Approved visual direction

The UI uses the existing 440 px compact window, 64 px sidebar, rounded shell, neutral charcoal palette, Tabler icons, 14 px minimum copy, light dividers, and no glow, purple accent, or unnecessary containers. The approved mockup shows a compact Project list and a Project detail with development actions, expected ports, and linked chats.
